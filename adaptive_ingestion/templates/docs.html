1. Architecture Overview

The dashboard presents a logical database view of the system. It hides all physical storage details and exposes only logical concepts.

2. Entities & Data Model

The system manages six logical entities for the IoT Smart Campus scenario:

users

Campus system users — students, staff, administrators.

user_id username email role

sensors

IoT sensors deployed across campus locations.

sensor_id user_id location sensor_type status

readings

Structured sensor readings represented as logical objects.

reading_id sensor_id record_id timestamp temperature humidity

sensor_events

Variable-payload sensor events — alerts, spikes, drops.

record_id event_type payload severity tags

logs

System and access logs with free-form JSON context.

record_id log_level source message context

alerts

Threshold-triggered alerts from sensors.

record_id alert_type triggered_by details resolved

3. Logical Object Identifier

Logical objects may include an identifier field (for example record_id) that the dashboard uses for update and delete operations.

4. Query JSON Format

All queries are submitted as JSON objects with the following structure:

{
  "operation":  "read" | "create" | "update" | "delete",
  "entity":     "entity_name",

  // ─── Read-specific fields ───
  "fields":     ["field1", "field2"],       // select specific fields
  "filters":    {"field": "value"},          // simple equality
  "conditions": [ ... ],                   // rich conditional filters
  "order_by":   "field_name",              // sort by field
  "order":      "asc" | "desc",            // sort direction
  "limit":      10,                        // max results

  // ─── Write-specific fields ───
  "data":       { ... }                     // for create / update
}

Required Fields

Field Required For Description
operation All One of: read, create, update, delete
entity All Logical entity name (e.g., users, readings)
data create, update Key-value pairs for the record
filters.record_id update, delete Identifies which record to modify

5. Filter Operators

Each condition in the conditions array is an object with three keys:

{"field": "field_name", "op": "operator", "value": value}
Operator Equivalent Description Example
eq = Equals {"field": "role", "op": "eq", "value": "admin"}
ne != Not equals {"field": "role", "op": "ne", "value": "student"}
gt > Greater than {"field": "temperature", "op": "gt", "value": 25}
gte >= Greater or equal {"field": "humidity", "op": "gte", "value": 60}
lt < Less than {"field": "temperature", "op": "lt", "value": 30}
lte <= Less or equal {"field": "humidity", "op": "lte", "value": 70}
like LIKE Pattern match (% = any chars, _ = one char) {"field": "username", "op": "like", "value": "%ali%"}
in IN (...) Set membership {"field": "role", "op": "in", "value": ["admin", "staff"]}

6. Logical Groups (AND / OR)

Default: AND

Multiple conditions in the top-level conditions array are combined with AND:

{
  "conditions": [
    {"field": "temperature", "op": "gte", "value": 22},
    {"field": "humidity",    "op": "lt",  "value": 65}
  ]
}
// → temperature >= 22 AND humidity < 65

Explicit OR

Wrap conditions in an {"or": [...]} object:

{
  "conditions": [
    {"or": [
      {"field": "sensor_type", "op": "eq", "value": "temperature"},
      {"field": "sensor_type", "op": "eq", "value": "humidity"}
    ]}
  ]
}
// → sensor_type = 'temperature' OR sensor_type = 'humidity'

Nesting

Groups can be nested arbitrarily:

{
  "conditions": [
    {"field": "temperature", "op": "gt", "value": 25},
    {"or": [
      {"field": "status", "op": "eq", "value": "active"},
      {"field": "status", "op": "eq", "value": "maintenance"}
    ]}
  ]
}
// → temperature > 25 AND (status = 'active' OR status = 'maintenance')

7. Sorting & Limiting

Field Type Description
order_by string Field name to sort by
order string "asc" (default) or "desc"
limit integer Maximum number of records to return 
{
  "operation": "read",
  "entity": "readings",
  "conditions": [{"field": "temperature", "op": "gt", "value": 25}],
  "order_by": "temperature",
  "order": "desc",
  "limit": 5
}

8. CRUD Operations

Create

Insert a new record. The system auto-generates record_id and routes fields to the correct backend.

{
  "operation": "create",
  "entity": "logs",
  "data": {
    "log_level": "INFO",
    "source": "dashboard",
    "message": "Manual entry",
    "context": {"user": "admin"}
  }
}

Read

Query records with optional field selection, conditions, sorting, and limiting.

{
  "operation": "read",
  "entity": "users",
  "fields": ["username", "role"],
  "conditions": [
    {"field": "role", "op": "ne", "value": "student"}
  ]
}

Update

Modify fields on an existing record (identified by record_id).

{
  "operation": "update",
  "entity": "sensor_events",
  "filters": {"record_id": "abc123"},
  "data": {
    "severity": "low"
  }
}

Delete

Remove a record from all backends that hold it (cascading).

{
  "operation": "delete",
  "entity": "logs",
  "filters": {"record_id": "abc123"}
}

9. Example Queries

{

    {"field": "temperature", "op": "gt", "value": 30}

  "fields": ["record_id", "temperature", "timestamp"]

10. REST API Endpoints

Method Endpoint Description
GET /api/entities List all logical entities with descriptions
GET /api/data/<entity> Read all records for an entity (supports query params as equality filters)
POST /api/data/<entity> Create a new record (JSON body)
PUT /api/data/<entity>/<record_id> Update a record (JSON body with patched fields)
DELETE /api/data/<entity>/<record_id> Delete a record from all backends
POST /api/query Execute a logical JSON query (full query format)
GET /api/session Get current session info
GET /api/schema View the logical schema (no backend details)
GET /api/query/history View query execution history
POST /api/acid-test/<name> Run ACID test (atomicity, consistency, isolation, durability, all)

11. Response Format

All API responses are JSON with the following structure:

Success (Read)

{
  "status": "ok",
  "count": 3,
  "data": [
    {"username": "alice", "role": "admin"},
    {"username": "bob",   "role": "staff"},
    {"username": "eve",   "role": "staff"}
  ]
}

Success (Create / Update / Delete)

{
  "status": "ok",
  "record_id": "a1b2c3d4e5f6"
}

Error

{
  "status": "error",
  "error": "Missing 'entity' field"
}

12. ACID Validation

Property What is tested How
Atomicity All-or-nothing across both databases Insert into the logical system, then attempt a conflicting write. Verify the overall operation is all-or-nothing.
Consistency Constraints are enforced Try inserting a reading with non-existent sensor_id. FK constraint blocks it.
Isolation Concurrent writes → consistent state Two threads update the same record simultaneously. Lock serialization prevents corruption.
Durability Data survives restarts Insert data, close all connections, reopen, verify data persists.

Run tests from the ACID Tests panel in the dashboard, or via CLI:

python3 transaction_manager.py
