Technical Documentation

HybridDB is an Advanced Hybrid Query Engine designed for high-performance data management across relational and document-based storage layers. It abstracts the complexity of heterogenous backends into a single, unified logical schema.

Core Philosophy: Decouple the logical application schema from physical storage constraints. Let the engine decide the optimal storage backend based on data characteristics.

System Architecture

The engine follows a multi-stage execution pipeline to process every query.

Client Request (JSON)
→
Query Analyzer & Router
→
Backend Parallel Execution
→
Temporal Result Merger

1. Analysis & Routing

The engine parses the JSON query and identifies which fields reside in SQL vs. NoSQL using a dynamic metadata registry.

2. Transaction Management

For write operations, the engine coordinates a distributed transaction, ensuring atomic updates across both SQLite and MongoDB.

Adaptive Data Routing

HybridDB uses a Schema-Aware Routing mechanism. Every field in your logical entity is mapped to a specific backend in metadata.json.

Field TypeBackend ChoiceLogic
Relational / High-ConsistencySQLite (SQL)Fields like username, spo2, and purchase_value are stored in relational tables.
Document / High-VelocityMongoDB (NoSQL)Fields like heart_rate, gps_lon, and comments are stored as JSON documents.
System MetadataSharedThe sys_ingested_at key is mirrored across both backends.

Temporal Merging

Since data for a single logical record is split across two databases, HybridDB uses a synthetic primary key called sys_ingested_at (ISO-8601 Timestamp with UTC).

How it works:

  1. Insertion: A single timestamp is generated at the start of the request.
  2. Distribution: The same timestamp is written to the SQL table and the Mongo document.
  3. Reconstruction: During a read, the engine performs a "Temporal Join", aligning rows from SQL with documents from Mongo based on this key.

ACID Validation Internals

Maintaining ACID properties in a hybrid environment requires sophisticated coordination.

Atomicity

Uses BEGIN IMMEDIATE in SQL. If a Mongo write fails, the engine triggers a Manual Compensation Logic to delete the partially inserted Mongo document and roll back the SQL transaction.

Isolation

Ensures that concurrent clients don't see "half-merged" records. The engine serializes writes to the relational layer to maintain a consistent logical state.

Advanced JSON Syntax

Our query language supports complex logic, including the __trace flag for deep debugging.

{
  "operation": "read",
  "entity": "health_metrics",
  "fields": ["username", "spo2", "heart_rate", "city"],
  "conditions": [
    {
      "or": [
        {"field": "city", "op": "eq", "value": "New York"},
        {"field": "spo2", "op": "lt", "value": 90}
      ]
    }
  ],
  "__trace": true
}

REST API Reference

Directly interact with the HybridDB engine via our high-performance REST endpoints.

POST /api/query

Unified Query Execution

The primary entry point for all CRUD operations. Accepts a JSON payload defining the operation and target entity.

GET /api/schema

Schema Inspection

Returns the unified logical schema, hiding physical backend distributions while showing available fields.

GET /api/session

Session Metadata

Retrieve system health, active session IDs, and backend connection status.
